# 여행 상세 — 인라인 시간표 펼치기 설계

## 배경

`docs/superpowers/specs/2026-07-22-trip-grouping-design.md`의 "열린 질문" 항목: 여행 상세
페이지(`/trips/{id}`)의 "포함된 날짜" 목록은 지금 각 날짜마다 사진 수와 `/map?date=YYYY-MM-DD`
로 이동하는 링크만 보여준다. 그 날의 전체 시간표(사진·주변 업장·메모)를 보려면 지도 페이지로
나가야 한다.

이 설계는 그 링크를 **인라인 펼치기 토글**로 바꿔, 여행 상세 페이지를 벗어나지 않고 그 날의
전체 시간표를 바로 볼 수 있게 한다.

## 범위

- **포함**: 날짜 항목 인라인 펼치기(아코디언, 한 번에 하나만), 날짜 노트(제목·내용) 표시·수정,
  시간대별 행(시각·사진 장수·사진 썸네일·주변 업장·메모) 표시, 메모 인라인 저장.
- **제외**: 사진 확대 뷰어(좌우 이동·회전·삭제) — 지도 페이지 고유 기능으로 남긴다. 여러 날짜
  동시 펼침. 신규 백엔드 엔드포인트(기존 4개를 그대로 재사용한다).
- 이 변경은 **순수 프론트엔드**다 — `app/Controllers/TripController.php`, 백엔드 서비스,
  DB 스키마는 건드리지 않는다.

## UI 구조

`app/Views/trip-detail.php`의 `renderDayList()`가 만드는 `<li>` 안의 "이 날 시간표 보기"
`<a>` 링크를 **토글 버튼**으로 교체한다. 클릭 시:

1. 이미 열려 있는 다른 날짜의 패널이 있으면 먼저 닫는다(아코디언 — 한 번에 하나만 열림).
2. 이 날짜를 처음 여는 것이면 `GET /timeline/{date}`를 호출해 그 `<li>` 바로 아래(형제 요소)에
   삽입한 패널에 렌더링한다. 이미 한 번 연 날짜를 다시 열 때는 캐시(메모리에 보관한 마지막 응답)를
   재사용해 재조회하지 않는다 — 단, 그 사이에 같은 패널에서 메모·노트를 저장했다면 저장 시점에
   캐시를 갱신해 반영한다.
3. 패널 안에는 `app/Views/map.php`의 시간표 레이어와 **동일한 내용**을 넣는다:
   - 날짜 노트(제목 input·내용 textarea·저장 버튼) — `day_notes` 테이블의 **그 날짜** 노트다.
     여행 상세 페이지 상단의 여행 제목·설명(트립 자체의 `title`/`body`, 날짜 범위 단위)과는
     별개의 데이터이므로 혼동하지 않도록 라벨을 "이 날의 제목/메모"로 둔다(트립 편집 폼의
     "제목"/"설명"과 시각적으로 구분되는 카드 스타일 사용).
   - 시간대별 행: 시각 라벨, 사진 장수, 사진 썸네일 그리드, 주변 업장 텍스트(있으면), 메모
     입력+저장 버튼.

**사진은 클릭해도 확대되지 않는다**(썸네일 표시만) — 확대 뷰어·회전·삭제는 지도 페이지 전용으로
남겨둔다(범위 제외 항목).

## 데이터 흐름

- `GET /timeline/{date}` 응답(`{date, day_note, slots}`)을 그대로 사용한다. 응답 형태는
  `TimelineService::buildForDate()`가 이미 만들어주는 것과 동일(`TimelineController.php`,
  `2026-07-22-trip-grouping-design.md` 작성 시점까지의 기존 구현).
- 슬롯에 좌표(`lat`/`lng`)가 있으면 사진 로드 완료를 기다린 뒤 `GET /timeline/poi?lat=&lng=`를
  **슬롯마다 순차** 호출한다 — `map.php`의 "사진 먼저, POI는 그 다음 순차 조회" 원칙을 그대로
  따른다(병렬 호출 금지, 세션 잠금·외부 API 부하 방지).
- 메모 저장: `POST /timeline/time-note`(`date`, `slot`, `memo`).
- 날짜 노트 저장: `POST /timeline/day-note`(`date`, `title`, `body`).
- 레이트리밋: `poi`(300/시간)·`notes`(120/시간) 버킷은 지도 페이지와 **공유**한다. 이 페이지
  전용 버킷을 새로 만들지 않는다 — 같은 사용자가 지도·여행 페이지를 오가며 쓰는 것이므로 합산
  한도가 자연스럽다.

## 에러 처리

모두 `map.php`의 기존 처리 패턴을 그대로 재사용한다.

- `GET /timeline/{date}` 실패 → 패널에 "시간별 동선을 불러오지 못했습니다." 표시.
- 슬롯별 `GET /timeline/poi` 실패 → 그 업장 표시 요소만 조용히 제거(패널 전체에는 영향 없음).
- `POST /timeline/time-note` 또는 `/timeline/day-note` 실패 → 저장 버튼에 "저장 실패"를
  잠깐 표시한 뒤 "저장"으로 복귀.

## 테스트 전략

이번 변경은 **신규 백엔드 코드가 전혀 없는 순수 프론트엔드 작업**이다(기존 `/timeline/{date}`,
`/timeline/poi`, `/timeline/day-note`, `/timeline/time-note`를 그대로 재사용). 따라서:

- `tests/feature/TripControllerTest.php::testShowRendersPageShell`에 인라인 확장에 필요한
  핵심 DOM 훅(토글 버튼 클래스·패널 컨테이너 등)이 렌더된 마크업에 존재하는지 어서션을 추가한다
  (`RouteControllerTest::testMapPageIncludesTimelineLayer`와 같은 패턴).
- 펼침/접힘·데이터 렌더링·메모 저장 같은 실제 동작은 이 프로젝트가 지금까지 해온 대로
  **브라우저 실측**으로 검증한다 — 프론트 JS 동작 자체를 자동 테스트하는 하네스가 이
  프로젝트에 없고, 기존 시간표 레이어도 같은 방식으로 검증돼 왔다.

## 접근안 비교(참고)

- **A. `map.php`의 시간표 렌더링 JS를 다듬어 이식(채택)** — `buildSlotRow`/`loadPoi`/`postNote`
  패턴을 인라인 삽입 형태로 재사용. 백엔드 변경 없음.
- **B. 공용 JS 파일로 분리해 두 페이지가 `<script src>` 공유** — 더 DRY하지만, 이 프로젝트는
  페이지마다 독립 인라인 스크립트 구조를 유지해왔고(SNS 공유 메뉴 JS도 같은 이유로 페이지마다
  중복시킴), 지금 이 구조를 깨는 건 이번 작업 범위를 넘어서는 변경이라 제외.
- **C. 서버에서 POI까지 합쳐 반환하는 새 엔드포인트** — 초기 응답이 외부 API(Overpass) 완료를
  기다려야 해 느려지고, "사진 먼저·POI 나중" 원칙(#26에서 확정)과 충돌해 제외.
